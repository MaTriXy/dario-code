<feature>
  <name>Hook Config Normalization, New Fields, and Deduplication</name>
  <files>src/core/hooks.mjs, tests/hooks-migration.test.mjs</files>
  <behavior>
    HOOK-01 — Format normalization:
    - normalizeHookConfig([{ matcher: "Bash", command: ["./test.sh"], timeout: 5000 }])
      → [{ matcher: "Bash", hooks: [{ type: "command", command: ["./test.sh"], timeout: 5000, statusMessage: null, once: false }] }]
    - normalizeHookConfig([{ matcher: "Bash", hooks: [{ command: "./test.sh" }] }])
      → [{ matcher: "Bash", hooks: [{ type: "command", command: ["./test.sh"], statusMessage: null, once: false }] }]
    - Mixed array with both flat and nested entries normalizes correctly
    - String command values are wrapped in array: "cmd" → ["cmd"]
    - Empty/null input returns []

    HOOK-02 — Type defaulting:
    - Handler without type field gets type: "command"
    - Handler with explicit type: "command" keeps it
    - Handler with type: "http" keeps it (forward compat for Phase 3)

    HOOK-03 — statusMessage:
    - Handler with statusMessage: "Checking..." preserves the value through normalization
    - statusMessage is available in execution context (passed to executeHook/dispatchHook)
    - Default statusMessage is null

    HOOK-04 — once per session:
    - Hook with once: true executes on first invocation, skipped on subsequent calls
    - Hook with once: false (or omitted) executes every time
    - once state is tracked at module level, survives across runHooks calls
    - clearOnceState() resets tracking (for tests and session end)

    HOOK-05 — Deduplication:
    - Two handlers with same type + command deduplicate to one
    - deduplicateHandlers([{type:"command",command:["./x.sh"]},{type:"command",command:["./x.sh"]}]) → [{type:"command",command:["./x.sh"]}]
    - Handlers with different commands are NOT deduplicated
    - Dedup happens AFTER matcher filtering (per research pitfall 3)
  </behavior>
  <implementation>
    1. RED phase: Create tests/hooks-migration.test.mjs with test cases for all 5 behaviors above. Mock loadSettings via vi.mock to inject hook configs. Run tests — all must fail.

    2. GREEN phase: Modify src/core/hooks.mjs:
       a. Add normalizeHookConfig(hookList) function — converts flat entries to nested, ensures type defaults to "command", wraps string commands in arrays, sets statusMessage/once defaults
       b. Add deduplicateHandlers(handlers) function — dedup by type + JSON.stringify(command)
       c. Add module-level _onceTracker = new Set() and helper functions hasRunOnce/markAsRun/clearOnceState
       d. Add dispatchHook(handler, context, verbose) function — strategy dispatch (only "command" type for now, calls existing executeHook logic)
       e. Update loadHooks() to normalize: return normalizeHookConfig(settings.hooks?.[eventType]) per event type, or normalize the entire hooks object
       f. Update runHooks() to: get normalized hooks, filter by matcher, collect handlers, dedup, filter once, dispatch each handler
       g. Pass handler.statusMessage through to execution context (available in result)
       h. Export normalizeHookConfig, deduplicateHandlers, clearOnceState for testing

    3. REFACTOR phase: Ensure all existing exports still work. Verify createHook still produces valid (flat) configs that normalize correctly. Run full test suite.
  </implementation>
